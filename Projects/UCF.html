Pros and cons of this configuration
Pros
  • Fits the 80-hour Tick budget without compromising essential Canvas elements.
  • No high scope-risk components (no Custom Illustrations, no Infographics).
  • No partner-dependency flags (no Avatar requiring headshot pipeline).
  • 3-button homepage navigation is simpler and lower-hours than a full Homepage component.
  • Footer, Line Break, and Section Banner provide structural clarity without per-course customization.
Cons
  • No Course Card means Canvas dashboard cards rely on default Canvas behavior.
  • No Module Banners means modules lack additional visual signaling.
  • 3 buttons are less rich than a full Canvas Homepage, narrower visual real estate.
Learner experience benefits (grounded in research)
  • Consistency reduces extraneous cognitive load. Generic components (Footer, Line Break, Section Banner) repeat the same visual cues across all 17 courses. Per Cognitive Load Theory, consistent visual language frees working memory for content learning.
  • Course-specific Course Banner supports Mayer's signaling principle. Per-course identity helps learners orient themselves to which course they're in across a multi-course program.
  • Homepage navigation buttons cluster related controls. Per Mayer's spatial contiguity principle, grouping homepage navigation in one visual block reduces wayfinding cost.
  • Visual hierarchy preserved. Section Banners and Line Breaks support hierarchical reading patterns established by visual hierarchy research.
Risk and dependency flags
Scope-risk flags None  No Custom Illustrations or Infographics in scope.
Partner-dependency flags None  No Avatar requiring headshot pipeline.
Substitution applied
Canvas Homepage component → 3 Ad-hoc homepage navigation buttons. Per the canon's substitution rule, an asset can be swapped for one with similar hour cost to flex the package mix. UCF's case substituted the higher-hour Homepage for lower-hour Ad-hoc buttons to fit the 80-hour budget while meeting design intent.
Alternative engine outputs

The engine produced several viable configurations for UCF's inputs. The team selected the chosen package above. These were the runners-up:

Maximum coverage configuration (Homepage + Course Banner + Course Card + Icon Set + Line Break + Footer + Module Banners + Avatar)
Why not chosen: would have exceeded the 80-hour Tick budget. Homepage and Course Card added Templated work UCF's Canvas setup did not require. Module Banners and Avatar added hours without proportional value.
Balanced configuration (Course Banner + Course Card + Icon Set + Line Break + Footer + Module Banners)
Why not chosen: Module Banners and Course Card added per-course and per-template hours UCF didn't need. The 3-button homepage navigation served the homepage role at lower cost.
Minimal effective configuration (Course Banner + Course Card + Icon Set)
Why not chosen: too thin. Missing the structural Generic components (Footer, Line Break, Section Banner) that UCF needed for a complete course experience.
Design Decisions Log Locked
No brand team access from UCF
UCF did not grant Creative access to a brand team during Program Planning. Brand foundations were developed using available partner branding references rather than direct partner brand authority collaboration. This shaped the package and surfaced a downstream risk that played out later in the project.

Custom homepage navigation: 3 buttons in place of standard Homepage
Substituted a 3-button homepage navigation pattern for the canon's standard Canvas Homepage component. Configuration agreed upon during Program Planning to fit UCF's Canvas setup and 80-hour budget.
Program Planning
Post-approval banner revision request declined
After Program Planning closed, a UCF admin requested revisions to the Course Banner. The decision was to move forward with the originally approved banner delivery due to scope. Partner ended up very happy with the banners in the end. This is the canonical post-approval change request pattern (FAQ entry 1) playing out in real life.
